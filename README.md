# Лабораторная работа №3: Lakehouse на Polars + Delta Lake

**Тема:** Построение пайплайна (bronze → silver → gold) для прогнозирования задержек авиарейсов на датасете US Flight Delays 2018–2024.

* **Оркестрация & Инфраструктура:** Docker Compose
* **Хранилище (Lakehouse):** MinIO (S3-совместимое) + Delta Lake
* **Вычислительный движок:** Polars (Lazy API)
* **ML Tracking:** MLflow
* **Машинное обучение:** Scikit-learn (RandomForest)

## Запуск проекта
Проект настроен на полный запуск одной командой. Приложение (app) и инфраструктура поднимаются автоматически, после чего оркестратор последовательно выполняет все этапы.

- Поместите исходные CSV-файлы в папку data/raw/

- Запустите инфраструктуру и пайплайн:

*docker compose up --build*

После успешного завершения пайплайна сервисы MLflow (http://localhost:5000) и MinIO (http://localhost:9001) продолжат работу в фоновом режиме для просмотра результатов.

## Обоснование выбора партиций (Silver)
Данные в слое Silver партицированы по ключам "year", "month"

Аналитические и ML-запросы к данным об авиарейсах чаще всего агрегируются или фильтруются по временным окнам (анализ трендов, сезонности, отчетность за конкретный месяц). Такое партицирование позволяет Delta Lake использовать Partition Pruning. При запросе данных за конкретный период, Polars обращается только к целевой директории на диске (S3), полностью игнорируя гигабайты ненужных файлов за другие годы. Это снижает нагрузку на сеть/диск и ускоряет Lazy-вычисления.

## Polars Lazy API: Физический план запроса (.explain)
В коде используется pl.scan_delta(), что позволяет Polars оптимизировать граф вычислений до начала физического чтения файлов.

Ниже представлен план запроса, сгенерированный при формировании слоя Silver:

simple π 15/15 ["FlightDate", "Airline", ... 13 other columns]
    WITH_COLUMNS: ...
        SELECT [col("FlightDate"), col("IATA_Code_Marketing_Airline").alias("Airline"), col("Origin"), col("Dest"), col("CRSDepTime"), col("DepDelay"), col("ArrDelay")]
            Parquet SCAN [s3://lakehouse/bronze/flights/part-00000-...]
            PROJECT 8/120 COLUMNS
            SELECTION: [([(col("Cancelled")) == (0.0)]) | ([([(col("Cancelled")) == (0.0)]) & (col("ArrDelay").is_not_null())])]

Здесь мы видим, что из 120 исходных колонок CSV-файла Polars физически считывает из Delta файлов только 8 необходимых столбцов. Это очень большая экономия оперативной памяти. Также, движок применяет фильтр отмененных рейсов на уровне чтения Parquet SCAN. Записи об отмененных рейсах даже не попадают в память, что ускоряет обработку датасета.

## Использованные возможности Delta Lake
Реализовано 5 возможностей Delta Lake сверх обязательного MERGE:

- MERGE: Выполняется в слое Silver. Повторный запуск пайплайна не дублирует данные, а обновляет их на основе генерируемого ключа flight_id.

- Schema Evolution: При загрузке файлов в слой Bronze используется опция schema_mode="merge" для безопасной адаптации к возможным изменениям в структуре CSV-файлов разных лет.

- Time Travel: В ML-пайплайне программно фиксируется версия таблицы dt.version(), на которой обучается модель. Эта версия логируется в MLflow для обеспечения воспроизводимости экспериментов.

- OPTIMIZE + Z-ORDER: В скрипте 05_optimize.py выполняется компактизация мелких файлов и кластеризация данных по колонке Origin для мгновенного выполнения запросов в разрезе аэропортов.

- VACUUM: Очистка устаревших транзакционных файлов в хранилище для освобождения места на S3.
